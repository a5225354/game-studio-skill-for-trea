# 设计文档规则 (Design Docs Rules)

每个 `design/gdd/` 中的设计文档必须包含以下 8 个必填章节：

1. **概述**：一段式总结，新团队成员也能理解
2. **玩家幻想**：玩家参与该机制时的感受和想象
3. **详细规则**：无歧义的机制描述。两个设计师读完后应构建出同样的东西
4. **公式**：所有数学公式及变量定义、输入范围和示例计算
5. **边界情况**：异常或极端情况下会发生什么 — 必须明确解决，不能是 "TBD"
6. **依赖关系**：列出与其他系统的双向交互和数据流
7. **调优旋钮**：可配置数值、预期范围、类别（手感/曲线/关卡）
8. **验收标准**：如何判断此机制是否正常工作 — 具体、可衡量

- 公式必须格式化为代码块（`result = base * multiplier`）
- 边界情况不能留空或用 "待定" 填充
- 依赖项必须明确系统名称，不能模糊表述
