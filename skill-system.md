### 1.3 核心差异

| 维度 | Claude Code 原方案 | Trae 改造方案 |
|------|--------------------|---------------|
| **入口加载** | CLAUDE.md 自动注入系统提示词 | 用户通过 Skill 工具调用 |
| **角色系统** | Task 独立子代理（独立上下文、独立模型） | 读取角色文件，主上下文内扮演 |
| **工作流触发** | 斜杠命令 `/xxx` + argument-hint | 自然语言关键词匹配 |
| **规则执行** | YAML paths 匹配 → 自动注入 | 提示词中明确指引，AI 主动遵循 |
| **钩子机制** | settings.json 事件驱动自动执行 | 手动 PS 脚本 + 技能内嵌检查 |
| **并行执行** | Task 工具并行调用多个子代理 | 顺序阶段切换 |
| **上下文管理** | PreCompact 钩子 + /compact 命令 | 文件持久化 + TodoWrite |

## 二、协作设计原则

### 2.1 核心理念

**用户驱动的协作，而非 AI 自主执行。**

```
正确模式：协作顾问             错误模式：自主生成器
═══════════════════            ═══════════════════
AI = 专家顾问                  ❌ AI 自行创建设计
用户 = 创意总监                 ❌ AI 未经讨论做决策
                                ❌ AI 不经批准写代码
AI 的职责：                    ❌ 用户只需说"做个游戏"
- 提出澄清问题
- 调研并呈现选项
- 解释权衡和推理过程
- 起草提案供审阅
- 写入前等待批准
```

### 2.2 协作五步法

```
1. 提问 → 代理主动提出澄清问题
   "为了设计制作系统，我需要了解：
    - 配方应该通过发现获得还是从 NPC 处学习？
    - 失败的尝试应该有多大的惩罚？"

2. 选项 → 展示 2-4 个方案及优缺点
   "基于您的目标，这里有三种方案：
    方案A：[描述] + 优缺点 + 参考游戏 + 支柱对齐
    方案B：[描述] + 优缺点 + 参考游戏 + 支柱对齐
    方案C：[描述] + 优缺点 + 参考游戏 + 支柱对齐"

3. 决策 → 用户做出选择
   使用 AskUserQuestion 工具呈现结构化选择界面

4. 草稿 → 基于决策起草内容
   逐章节起草，展示草稿或摘要
   对歧义处询问，不自行假设

5. 批准 → 写入前获得明确确认
   "我可以将此写入 [文件路径] 吗？"
   用户说"可以"后才使用 Write/Edit 工具
```

### 2.3 好的提问模式

✅ **带推理的多选题**:
```
"敌人是否应该有攻击预警？
 A) 是的，0.5秒前（可及性强，节奏导向）
 B) 是的，0.2秒前（时机紧凑，技能导向）
 C) 没有预警（纯模式学习，高难度）
 哪种符合你的愿景？"
```

✅ **带权衡的约束选项**:
```
"背包系统选项：
 1. 网格式：空间管理深度高，速度较慢
 2. 列表式：快速访问，策略性较低
 3. 混合式：中等复杂度
 鉴于你的'有意义的选择'支柱，我倾向于1或3。你怎么看？"
```

### 2.4 错误的提问模式

❌ 过于开放: "战斗系统应该是什么样的？"
❌ 引导/预设: "我将把战斗做成实时的，因为这是标准做法。"
❌ 无上下文的二选一: "要不要做技能树？是或否？"

### 2.5 AskUserQuestion 使用指南

✅ **适用于**:
- 每个需要展示 2-4 个选项的决策点
- 答案受约束的初始澄清问题
- 将最多 4 个独立问题批量处理在一次调用中
- 下一步选择、架构决策、战略选择

❌ **不适用于**:
- 开放式探索问题
- 单个是/否确认
- 对话流程中自然推进的步骤

## 三、意图路由机制

### 3.1 路由流程

```
用户输入: "我想设计一个战斗系统"
        │
        ▼
    [关键词提取]
    "设计" + "战斗" + "系统"
        │
        ▼
    [路由表匹配]
    匹配到: skills/design-system.md
        │
        ▼
    [读取工作流文件]
    Read: skills/design-system.md
        │
        ▼
    [按工作流执行]
    1. 检测 → 2. 询问 → 3. 提供选项 → 4. 起草 → 5. 批准
```

### 3.2 路由优先级

当用户意图匹配多个路由时，按以下优先级：
1. 团队编排 (team-combat 等) — 最明确
2. 发布与运营 (release-checklist 等) — 具体流程
3. 项目管理 (sprint-plan 等) — 管理动作
4. 开发实现 (code-review 等) — 编码动作
5. 创意设计 (brainstorm 等) — 创意动作
6. 入门 (start 等) — 兜底

### 3.3 模糊意图处理

当用户意图不够明确时，使用 AskUserQuestion 让用户细化：

```
用户: "帮我做个游戏"
        ↓
检测到模糊意图，使用 AskUserQuestion:
  Q: "你目前处于什么阶段？"
  A) 还没有想法 — 从头脑风暴开始
  B) 有模糊想法 — 发展概念
  C) 有清晰概念 — 正式化文档
  D) 已有成果 — 整理和继续
```

## 四、角色扮演机制

### 4.1 角色激活流程

```
触发条件: 工作流需要特定角色知识
        │
        ▼
读取 agents/[角色名].md
        │
        ▼
在当前上下文中扮演该角色:
  - 遵循角色的核心职责
  - 使用角色的理论框架
  - 遵守角色的协作协议
  - 参考角色的委派/上报关系
        │
        ▼
完成角色任务后切换回协调者或下一角色
```

### 4.2 角色文件结构

每个角色文件包含以下部分：

```markdown
# 角色: [角色名]

## 角色定位
- 层级: 第X层
- 上级: [角色名]
- 下级: [角色列表]
- 协作方: [角色列表]

## 激活方式
[何时激活此角色]

## 核心职责
[该角色的具体职责]

## 理论框架
[该角色使用的专业理论和框架]

## 工作标准
[该角色的产出质量要求]

## 协作协议
[该角色如何与用户协作]

## 委派关系
[该角色可以向谁委派，接受谁的上报]
```

### 4.3 多角色切换

```
Phase 1: 激活 game-designer → 完成设计
  [TodoWrite: ✅ 阶段1完成]
        ↓
Phase 2: 激活 gameplay-programmer → 完成架构
  [TodoWrite: ✅ 阶段2完成]
        ↓
Phase 3: 激活 qa-tester → 完成测试
  [TodoWrite: ✅ 阶段3完成]
```

每次切换使用 TodoWrite 跟踪进度。

## 五、上下文管理策略

### 5.1 文件持久化

**文件即记忆，而非对话。** 对话是短暂的；磁盘上的文件在会话间持久存在。

关键文件：
- `production/session-state/active.md` — 活跃检查点
- 各阶段产出的设计文档/代码文件

状态文件应包含：
- 当前任务和进度
- 已做出的关键决策
- 正在处理的文件
- 待解答的问题

### 5.2 增量文件写入

创建多章节文档时：
1. 立即创建文件骨架（所有章节标题，空正文）
2. 在对话中逐节讨论和起草
3. 每节获批后立即写入文件
4. 每节完成后更新会话状态文件

**为什么**: 完整的设计文档会话可能积累大量上下文。增量写入将实时上下文保持在仅当前章节的讨论范围内，因为已完成的章节已持久化到磁盘。

### 5.3 上下文恢复

重新开始会话时：
1. 读取 `production/session-state/active.md`
2. 读取状态中列出的未完成文件
3. 从下一个未完成的章节或任务继续

## 七、与原Claude Code方案的关键差异总结

### 7.1 不需要用户操作的变化

| 维度 | 原方案 | 新方案 |
|------|--------|--------|
| 调用方式 | Claude Code内自动加载 | 用户在Trae中调用 Skill("game-studio") |
| 命令触发 | `/brainstorm roguelike` | "帮我头脑风暴一个Roguelike概念" |
| 参数传递 | `/sprint-plan new` | "创建新冲刺" 或 "规划冲刺" |

### 7.2 需要用户手动执行的操作

| 操作 | 原方案 | 新方案 |
|------|--------|--------|
| 会话开始时获取上下文 | session-start.sh 自动运行 | 运行 `.\hooks\session-start.ps1` 或AI帮你运行 |
| 提交前验证 | validate-commit.sh 自动运行 | AI在code-review工作流中提醒检查 |
| 资源写入后验证 | validate-assets.sh 自动运行 | AI在相关技能中主动检查 |
| 推送前保护 | validate-push.sh 自动运行 | 由用户自行决定是否运行 |

### 7.3 性能影响

| 维度 | 原方案 | 新方案 |
|------|--------|--------|
| 角色切换 | Task子代理独立上下文，无性能影响 | 角色指令在主线上下文中，占用上下文空间 |
| 并行执行 | ✓ 真正并行 | ✗ 只能顺序执行 |
| 自动检查 | 事件触发零延迟 | AI需在提示词中主动执行，可能遗漏 |
