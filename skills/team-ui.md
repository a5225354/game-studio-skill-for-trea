# Team UI — UI团队编排技能

触发条件:
- 用户说"UI团队" / "team ui" / "界面" / "UI功能"
- 用户提供了UI功能描述，如 "用UI团队设计主菜单"

---

当用户触发此技能时，通过结构化流水线编排 UI 开发。采用**顺序阶段角色扮演**模式。

**决策点**: 在每个阶段转换时，使用 `AskUserQuestion` 将方案作为可选项呈现给用户。用户批准后方可进入下一阶段。

## 团队组成
- **ux-designer** — 用户流程、线框图、无障碍访问、输入处理
- **ui-programmer** — UI 框架、屏幕、控件、数据绑定、实现
- **art-director** — 视觉风格、布局打磨、与美术圣经的一致性

## 角色激活方式

在每个阶段，读取对应的角色文件并扮演该角色：
- `agents/ux-designer.md` → UX 设计
- `agents/art-director.md` → 视觉方向
- `agents/ui-programmer.md` → UI 实现

使用 `TodoWrite` 跟踪各阶段进度。

## 流水线

### 阶段 1：UX 设计
激活 **ux-designer** 角色：
- 定义此功能的用户流程（入口点、状态、出口点）
- 为每个屏幕/状态创建线框图（用文字描述布局）
- 规定交互模式：键盘/鼠标和游戏手柄如何导航此界面？
- 定义无障碍要求：文字大小、对比度、色盲安全
- 识别 UI 需要展示的数据（读取哪些游戏状态？）
- 使用 AskUserQuestion 呈现流程选项
- 输出：包含线框图和交互映射的 UX 规格

### 阶段 2：视觉设计
激活 **art-director** 角色：
- 对照美术圣经审查线框图
- 定义视觉处理方案：颜色、字体、间距、动画
- 规定资产需求（图标、背景、装饰元素）
- 确保与现有 UI 屏幕的一致性
- 输出：包含样式说明的视觉设计规格

### 阶段 3：实现
激活 **ui-programmer** 角色：
- 按照 UX 规格和视觉设计实现 UI
- 确保 UI 绝不拥有或修改游戏状态 — 仅展示，通过事件请求操作
- 所有文本通过本地化系统 — 禁止硬编码字符串
- 同时支持两种输入方式（键盘/鼠标 + 游戏手柄）
- 实现无障碍功能（文字缩放、色盲模式支持）
- 连接数据绑定到游戏状态
- 输出：已实现的 UI 功能

### 阶段 4：审查（顺序执行）
先激活 **ux-designer** 角色：
- 验证实现是否符合线框图和交互规格
- 测试纯键盘和纯手柄导航
- 检查无障碍访问

再激活 **art-director** 角色：
- 验证与美术圣经的视觉一致性
- 在最低和最高支持分辨率下进行检查

### 阶段 5：打磨
- 处理审查反馈
- 验证动画可跳过并尊重运动偏好设置
- 确认 UI 音效通过音频事件系统触发
- 在所有支持的分辨率和宽高比下进行测试

## 输出
一份总结报告，涵盖：UX 规格状态、视觉设计状态、实现状态、无障碍合规性、输入方式支持情况，以及任何未解决的问题。

## 协作协议
1. 每阶段开始前读取对应角色文件
2. 使用 AskUserQuestion 呈现关键决策选项
3. 使用 TodoWrite 跟踪进度
4. 写入任何文件前获得用户批准
