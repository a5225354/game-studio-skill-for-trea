# Create Architecture — 主架构文档

本技能生成 `docs/architecture/architecture.md` — 将所有已批准 GDD 翻译为具体技术蓝图的主架构文档。它位于设计和实现之间，在冲刺计划开始前必须存在。

**不同于 architecture-decision**: ADR 记录个别决策点。本技能创建给 ADR 提供上下文的完整系统蓝图。

---

## 触发条件

- 用户说"创建架构" / "设计架构" / "create architecture" / "主架构文档"

**参数模式:**
- **无参数 / `full`**: 完整引导式逐章节编写
- **`layers`**: 仅系统分层图
- **`data-flow`**: 仅模块间数据流
- **`api-boundaries`**: 仅 API 边界定义
- **`adr-audit`**: 仅审计现有 ADR 的引擎兼容性缺口

---

## 阶段0: 加载全部上下文

### 0a. 引擎上下文（关键）
Read 引擎参考库完全加载：VERSION.md、breaking-changes.md、deprecated-apis.md、current-best-practices.md 及所有 `modules/` 文件。

如果引擎未配置，停止并提示先运行 setup-engine。

### 0b. 设计上下文 + 技术需求提取
Read 所有已批准设计文档并从每个提取技术需求。

构建**技术需求基线** — 跨所有 GDD 提取的所有需求的平面列表，编号 `TR-[gdd-slug]-[NNN]`。

### 0c. 现有架构决策
Read `docs/architecture/` 中所有文件。

### 0d. 生成知识缺口清单
展示结构化摘要，包含 HIGH/MEDIUM/LOW 风险域。使用 AskUserQuestion 询问如何继续。

---

## 阶段1: 系统分层映射

将 `systems-index.md` 中每个系统映射到架构分层：

```
演示层 → 功能层 → 核心层 → 基础层 → 平台层
```

呈现图层分配方案并征求批准。

---

## 阶段2: 模块所有权映射

对阶段1中定义的每个模块，定义：拥有什么、暴露什么、消费什么、使用的引擎 API。

---

## 阶段3: 数据流

定义关键游戏场景中模块间数据移动方式：帧更新路径、事件/信号路径、存档/读档路径、初始化顺序。

---

## 阶段4: API 边界

定义模块间公共契约：接口、入口点、调用者必须遵守的不变量、模块必须保证的内容。

---

## 阶段5: ADR 审计 + 可追溯检查

审查所有现有 ADR 的质量和可追溯覆盖。列出需要的新 ADR。

---

## 阶段6: 缺失 ADR 列表

按优先级分组列出应存在但尚未创建的所有 ADR。

---

## 阶段7: 写入主架构文档

所有章节批准后，写入 `docs/architecture/architecture.md`。文档包含：文档状态、引擎知识缺口摘要、系统分层图、模块所有权、数据流、API 边界、ADR 审计、需要的 ADR、架构原则、开放问题。

---

## 阶段7b: 技术总监签收 + 首席程序员可行性审查

自审 + 可选扮演 lead-programmer 角色进行可行性审查。呈现两边评估结果并征求用户批准。

---

## 阶段8: 交接

更新会话状态，输出交接报告，列出需要运行的 ADR 列表和门禁就绪状态。

---

## 协作协议

1. **静默加载上下文** — 不叙述文件读取
2. **呈现发现** — 展示知识缺口清单和分层方案
3. **决策前询问** — 对每个架构选择呈现选项
4. **批准前起草** — 在询问写入前展示内容
5. **使用 AskUserQuestion 获取写入批准**
6. **增量写入** — 每个批准章节立即写入文件

绝不未经用户输入做出约束性架构决策。

Verdict: **COMPLETE** — 架构文档已写入。
