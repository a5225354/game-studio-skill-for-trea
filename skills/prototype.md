# Prototype — 快速原型验证技能

触发条件:
- 用户说"原型" / "prototype" / "验证" / "概念验证" / "快速验证"

---

当用户触发此技能时:

## 1. 明确核心问题

读取概念描述。识别此原型必须回答的核心问题。如果概念模糊，在继续之前用 AskUserQuestion 明确陈述问题。

## 2. 创建原型计划

用 3-5 个要点定义最小可行原型的样子。核心问题是什么？回答它所需的最少代码量是多少？什么可以跳过？

## 3. 创建原型目录

`prototypes/[concept-name]/`，其中 `[concept-name]` 是从概念派生的短 kebab-case 标识符。

## 4. 在隔离目录中实现原型

每个文件必须以以下内容开头：
```
// PROTOTYPE - NOT FOR PRODUCTION
// 问题：[正在测试的核心问题]
// 日期：[当前日期]
```

标准有意放宽：
- 可以随意硬编码值
- 使用占位资源
- 跳过错误处理
- 使用最简单可行的方案
- 复制代码而非从正式代码导入

## 5. 测试概念

运行原型。观察行为。收集可测量的数据（帧时间、交互次数、手感评估）。

## 6. 生成原型报告

保存到 `prototypes/[concept-name]/REPORT.md`：

```markdown
## 原型报告：[概念名称]

### 假设
[我们预期为真的内容 — 我们要回答的问题]

### 方法
[我们构建了什么、花了多长时间、走了哪些捷径]

### 结果
[实际发生了什么 — 具体观察]

### 指标
- 帧时间：[如相关]
- 手感评估：[具体 — "200ms 延迟时响应感觉迟钝"]
- 迭代次数：[尝试了多少次才成功]

### 建议：[推进 / 转向 / 放弃]
[用证据解释建议的一段话]

### 如果推进
[从原型到正式实现需要改变什么]

### 经验教训
[影响其他系统或未来工作的发现]
```

## 重要约束

- 原型代码**绝不能**从正式源文件导入
- 正式代码**绝不能**从原型目录导入
- 如果建议是推进，正式实现必须从头编写 — 原型代码不会被重构为正式代码
- 原型总工作量应控制在相当于 1-3 天的工作量内
- 如果原型范围开始膨胀，停下来重新评估问题是否可以简化

## 协作协议

1. 明确核心问题和假设
2. 创建最小原型计划
3. 实现原型代码
4. 生成结构化的原型报告
5. 写入任何文件前获得用户批准
