# Estimate — 工作量估算技能

触发条件:
- 用户说"估算" / "estimate" / "工作量" / "多久" / "预估"
- 用户提供了任务描述，如 "估算实现伤害计算需要多久"

---

当用户触发此技能时:

## 1. 读取任务描述
如果描述过于模糊以至于无法进行有意义的估算，使用 AskUserQuestion 要求澄清。

## 2. 读取项目上下文
读取 SKILL.md 了解技术栈和编码标准。读取相关设计文档（`design/gdd/`）了解任务关联的功能。

## 3. 扫描代码库
- 使用 Glob/Grep 识别需要变更的文件和模块
- 评估这些文件的复杂度（大小、依赖数量）
- 识别与其他系统的集成点

## 4. 读取过去的冲刺数据
如果 `production/sprints/` 中有历史数据:
- 查找类似的已完成任务及其实际工作量
- 计算历史速度（计划 vs 实际）
- 识别任何估算偏差模式

## 5. 分析以下因素

**代码复杂度**:
- 受影响文件的代码行数
- 依赖数量和耦合程度
- 是否涉及核心/引擎代码 vs 功能代码
- 是否可以遵循现有模式，还是需要新模式

**范围**:
- 涉及的系统数量
- 新代码 vs 修改现有代码的比例
- 需要的新测试覆盖量

**风险**:
- 新技术或不熟悉的库
- 不明确或模糊的需求
- 对未完成工作的依赖
- 跨系统集成复杂度

## 6. 生成估算

```markdown
## 任务估算：[任务名称]
生成时间：[日期]

### 任务描述
[用 1-2 句话清楚地重述任务]

### 复杂度评估

| 因素 | 评估 | 备注 |
|------|------|------|
| 受影响系统 | [列表] | [核心、玩法、UI 等] |
| 可能修改的文件 | [数量] | [关键文件列在下方] |
| 新代码 vs 修改 | [比例] | |
| 集成点 | [数量] | [哪些系统交互] |
| 需要的测试覆盖 | [低 / 中 / 高] | |

**可能受影响的关键文件：**
- `[path/to/file1]` -- [这里的变更内容]
- `[path/to/file2]` -- [这里的变更内容]

### 工作量估算

| 场景 | 天数 | 假设条件 |
|------|------|----------|
| 乐观 | [X] | 一切顺利，没有意外，需求明确 |
| 预期 | [Y] | 正常节奏，有少量问题 |
| 悲观 | [Z] | 出现重大未知因素，需求变更 |

**建议预算：[Y 天]**

### 置信水平：[高 / 中 / 低]

**高** -- 需求明确，系统熟悉，遵循现有模式，之前完成过类似任务。
**中** -- 存在一些未知因素，涉及中等复杂度的系统。
**低** -- 存在重大未知因素，新技术，需求不明确，或涉及跨多个系统。

### 风险因素

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|---------|
| [具体风险] | [高/中/低] | [如果发生则增加的天数] | [如何降低风险] |

### 依赖关系

| 依赖项 | 状态 | 延迟时的影响 |
|--------|------|-------------|
| [必须先完成的工作] | [已完成 / 进行中 / 未开始] | [如何影响此任务] |

### 建议拆分

| # | 子任务 | 估算 | 备注 |
|---|--------|------|------|
| 1 | [调研 / 探索性实验] | [X 天] | [如果未知因素需要先调查] |
| 2 | [核心实现] | [X 天] | [主要工作] |
| 3 | [集成] | [X 天] | [连接到现有代码] |
| 4 | [测试和验证] | [X 天] | [编写测试、手动验证] |
| 5 | [代码审查和迭代] | [X 天] | [审查反馈、修复] |
| | **合计** | **[Y 天]** | |

### 备注与假设
- [影响估算的关键假设]
- [关于范围边界的注意事项 — 包含什么 vs 排除什么]
```

## 7. 向用户输出估算
附带简短摘要：建议预算、置信水平和最大的单个风险因素。

## 估算指南

- **始终给出一个范围**（乐观 / 预期 / 悲观），绝不要只给一个数字。单点估算会造成虚假精确的错觉
- 建议预算应为预期估算，而非乐观估算。留有余地是务实的做法
- 如果置信水平为低，建议先进行限时探索性实验或原型验证
- 明确说明包含和排除的内容。范围模糊是估算错误最常见的原因
- 如果任务太大无法自信估算（预期超过 10 天），建议拆分为更小的子任务
- 不要暗中增加估算缓冲。如果存在风险，在风险因素部分明确指出

## 协作协议

1. 读取相关代码和文档上下文
2. 使用 AskUserQuestion 澄清模糊需求
3. 展示估算草稿，附带置信水平说明
4. 写入前获得用户批准
