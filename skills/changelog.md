# Changelog — 变更日志生成技能

触发条件:
- 用户说"变更日志" / "changelog" / "更新日志" / "内部更新日志"
- 用户提供了版本号，如 "生成 v1.2.0 变更日志"

---

当用户触发此技能时:

## 1. 确定版本

提取用户指定的版本号。如果未提供，使用 AskUserQuestion 询问。

## 2. 收集变更数据

从多个来源读取:
- Git 提交日志（从一个发布标签到当前标签之间）
- Sprint 计划和回顾（`production/sprints/`）
- 设计文档变更（`design/gdd/` 中最近修改的文件）
- Bug 修复记录

## 3. 分类所有变更

- **新功能（New Features）** — 新增的系统、机制、功能
- **改进（Improvements）** — 对现有功能的增强
- **Bug 修复（Bug Fixes）** — 按系统分组（战斗、UI、联网等）
- **性能（Performance）** — 优化和性能改进
- **技术变更（Technical）** — 内部重构、工具改进
- **移除（Removed）** — 已删除的功能或内容

## 4. 生成内部变更日志

```markdown
# 变更日志 — [版本]
发布日期：[日期]

## 新功能
- **[功能名称]**：[1-2句描述]（引用设计文档：[path]）

## 改进
- **[改进内容]**：[描述]（引用相关提交：[hash]）

## Bug 修复
### 战斗
- 修复了 [问题描述]（[缺陷ID]，提交：[hash]）

### UI
- 修复了 [问题描述]

## 性能
- [优化描述]（从 [旧值] 改善到 [新值]）

## 技术变更
- [内部重构/工具变更]

## 移除
- [已删除的功能及原因]

## 已知问题
- [问题及可用的临时解决方案]
```

## 5. 对比 patch-notes

内部 changelog 与面向玩家的 patch-notes 的区别：
- Changelog：技术化、面向团队、包含引用（设计文档/提交/缺陷ID）
- Patch-notes：玩家友好、翻译技术语言、聚焦玩家可感知的变更

## 协作协议

1. 扫描 Git 历史和生产文档
2. 分类和描述所有变更
3. 展示完整 changelog 草稿
4. 写入前获得用户批准
