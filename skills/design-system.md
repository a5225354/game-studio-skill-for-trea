# Design System — 引导式GDD编写技能

触发条件:
- 用户说"设计系统" / "设计战斗" / "设计机制" / "GDD" / "设计文档" / "design system"
- 用户提供具体系统名称，如"设计制作系统" / "设计背包系统"

参数提取:
- 如果用户提供了系统名称（如"combat-system"），将其作为设计目标
- 如果无参数，先询问用户要设计什么系统

---

## 协作协议

**严格遵循协作五步法：提问 → 选项 → 决策 → 草稿 → 批准**

使用增量章节写入模式:
1. 先创建文件骨架（所有 8 个章节标题，空正文）
2. 在对话中逐节讨论和起草
3. 每节获批后立即写入文件
4. 每节完成后更新进度

---

## 工作流

### 步骤 0: 确定设计目标

如果用户未指定系统名称，使用 AskUserQuestion 询问：
- 你想设计哪个游戏系统？
- 有没有已有的系统索引可以参考？(检查 `design/gdd/systems-index.md`)

如果系统索引存在且用户不确定，读取并展示未设计的系统列表。

### 步骤 1: 读取上下文

在开始设计前，读取:
1. `design/gdd/game-concept.md` — 获取游戏概念和支柱
2. `design/gdd/systems-index.md` — 获取系统依赖关系（如果存在）
3. 任何已有相关 GDD — 获取已设计的相邻系统

### 步骤 2: 提出澄清问题

在起草任何内容之前，基于游戏支柱和概念提出针对性的澄清问题：

示例（设计制作系统时）:
> "为了设计制作系统，我需要了解：
> - 配方应该通过实验发现还是从 NPC 处学习？
> - 失败的尝试应该有多大的惩罚？
> - 目标受众主要是探索者还是成就者？
> - 这与你的游戏支柱如何对齐？"

### 步骤 3: 提供方案选项

基于用户回答，提供 2-4 个机械方案：

每个方案包含:
- 核心机制描述
- 优点（带推理）
- 缺点（诚实陈述）
- 参考游戏
- 与游戏支柱的对齐评估

示例:
```
**方案 A：纯发现** — 无提示，完全自由
优点: 最大探索奖励
缺点: 高挫败风险，可能需要 wiki
参考: 《我的世界》早期

**方案 B：提示系统** — 材料兼容时显示视觉提示
优点: 引导式探索，较少挫败
缺点: 较少"顿悟"时刻

**方案 C：混合** — 自由实验 + 通过技能解锁的可选日志提示
优点: 支持两种游玩风格，渐进式脚手架
缺点: 实现更复杂
```

使用 AskUserQuestion 让用户选择方案。

### 步骤 4: 创建文件骨架

获得用户方案选择后，立即创建目标文件的骨架:

```
询问: "我可以创建 design/gdd/[系统名].md 并包含章节骨架吗？"
用户确认后 → 创建包含 8 个章节标题（空正文）的文件
```

### 步骤 5: 逐章节起草（增量写入）

按以下顺序逐节起草，每节获批后立即写入：

#### 第 1 节: 概述
一段式总结，新团队成员也能理解。包含：
- 系统是什么
- 为什么需要它
- 在游戏中的角色

#### 第 2 节: 玩家幻想
玩家参与该系统时的想象/感受。引用该系统主要服务的美学目标（MDA 框架中的 A）。
- 玩家应该有什么感受？
- 什么时刻让玩家感到满足？
- 与核心游戏支柱的关系？

#### 第 3 节: 详细规则
精确、无歧义的机制描述。程序员应能仅凭本节进行实现。
- 逐条列出所有规则
- 使用明确的条件/动作语言
- 避免模糊表述（"可能"、"有时"、"通常"）

#### 第 4 节: 公式
所有数学公式及变量定义：
- 公式: `result = base * multiplier + bonus`
- 变量定义和输入范围
- 示例计算（正常情况和极端情况）
- 对非线性曲线附带说明

#### 第 5 节: 边界情况
异常/极端情况下会发生什么：
- 最小值/最大值行为
- 除零/空值/溢出保护
- 与其他系统的异常交互
- 退化策略及其缓解措施

#### 第 6 节: 依赖关系
- 本系统依赖哪些其他系统（上游依赖）
- 哪些其他系统依赖本系统（下游依赖）
- 数据流方向
- 集成契约

#### 第 7 节: 调优旋钮
暴露用于平衡的可配置数值:
- **手感旋钮**: 影响逐刻体验（速度、时机）
- **曲线旋钮**: 影响成长形状（倍率、阈值）
- **关卡旋钮**: 影响节奏（冷却、资源限制）
- 每个旋钮: 预期范围、默认值、理由

#### 第 8 节: 验收标准
如何判断此系统正常工作:
- 功能标准（是否做了正确的事？）
- 体验标准（是否感觉对？试玩验证什么？）
- 可衡量的测试条件

### 步骤 6: 最终检查

全部 8 个章节完成并写入后:
1. 对照 8 章节标准自检完整性
2. 检查公式是否无歧义
3. 检查边界情况是否已明确（非 "TBD"）

然后告知用户:
> "完整的 GDD 草稿已完成。我已涵盖全部 8 个章节。
> 你希望我运行设计评审来验证其完整性吗？（读取 skills/design-review.md 执行）"

---

## GDD 质量要求

`design/gdd/` 中的每份文档必须满足:
1. ✅ 8 个必填章节齐全且完整
2. ✅ 公式无歧义，所有变量已定义
3. ✅ 边界情况已明确解决（不是"TBD"）
4. ✅ 依赖项列出了双向关系
5. ✅ 验收标准可测试且可衡量

---

## 护栏

- 绝不在未经批准的情况下写入文件
- 每节起草前先展示草稿或摘要
- 对边界情况中的歧义主动询问，不自行假设
- 如果设计涉及复杂数学，咨询 systems-designer 角色
- 如果设计涉及经济平衡，咨询 economy-designer 角色
