# Milestone Review — 里程碑评审技能

触发条件:
- 用户说"里程碑" / "milestone" / "检查点" / "里程碑评审" / "milestone review"
- 用户询问 "alpha 怎么样了" / "beta 进度" 等

---

当用户触发此技能时:

## 1. 读取里程碑定义
从 `production/milestones/` 获取。

## 2. 读取该里程碑内的所有冲刺报告
从 `production/sprints/` 中读取。

## 3. 扫描代码库中的标记
搜索 TODO、FIXME、HACK 标记，以识别未完成的工作。

## 4. 检查风险登记册
位于 `production/risk-register/`。

## 5. 生成里程碑审查

```markdown
# 里程碑审查：[里程碑名称]

## 概述
- **目标日期**：[日期]
- **当前日期**：[今日]
- **剩余天数**：[N]
- **已完成冲刺**：[X/Y]

## 功能完成度

### 完全完成
| 功能 | 验收标准 | 测试状态 |
|------|---------|---------|

### 部分完成
| 功能 | 完成百分比 | 剩余工作 | 对里程碑的风险 |
|------|-----------|---------|---------------|

### 未开始
| 功能 | 优先级 | 可否裁剪？ | 裁剪影响 |
|------|--------|-----------|---------|

## 质量指标
- **未修复的 S1 Bug**：[N] -- [列表]
- **未修复的 S2 Bug**：[N]
- **未修复的 S3 Bug**：[N]
- **测试覆盖率**：[X%]
- **性能**：[是否在预算内？详情]

## 代码健康度
- **TODO 数量**：[N]
- **FIXME 数量**：[N]
- **HACK 数量**：[N]
- **技术债务项**：[列出关键的]

## 风险评估
| 风险 | 状态 | 发生后的影响 | 缓解状态 |
|------|------|------------|---------|

## 速率分析
- **计划 vs 完成**（所有 Sprint）：[X/Y 任务 = Z%]
- **趋势**：[提升 / 稳定 / 下降]
- **剩余工作的调整估算**：[按当前速率所需天数]

## 范围建议
### 保护（必须随里程碑交付）
- [功能及原因]

### 有风险（可能需要裁剪或简化）
- [功能及风险]

### 裁剪候选（可推迟而不影响里程碑）
- [功能及裁剪影响]

## 推进/暂停评估

**建议**：[推进 / 有条件推进 / 暂停]

**条件**（如有条件）：
- [必须满足的条件 1]
- [必须满足的条件 2]

**理由**：[建议的解释]

## 行动项
| # | 行动 | 负责人 | 截止日期 |
|---|------|--------|---------|
```

## 6. 展示摘要

审查完成后，向用户展示关键发现:
- 功能完成百分比和趋势
- Go/No-Go 建议
- 最重要的 3 个行动项

写入前询问用户:
> "我可以将此里程碑审查写入 `production/milestones/[名称]-review.md` 吗？"

## 协作协议

1. 读取所有相关数据（里程碑、冲刺、风险登记册）
2. 客观分析，不美化数据
3. 展示完整审查草稿
4. 写入前获得用户批准
