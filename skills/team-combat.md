# Team Combat — 战斗团队编排技能

触发条件:
- 用户说"战斗团队" / "team combat" / "开发战斗" / "战斗功能"
- 用户提供了具体战斗功能描述，如 "用战斗团队开发钩爪能力"

---

当用户触发此技能时，通过结构化流水线编排战斗开发。采用**顺序阶段角色扮演**模式。

**决策点**: 在每个阶段转换时，使用 `AskUserQuestion` 将方案作为可选项呈现给用户。用户批准后方可进入下一阶段。

## 团队组成
- **game-designer（游戏设计师）** — 设计机制，定义公式和边缘情况
- **gameplay-programmer（玩法程序员）** — 实现核心玩法代码
- **ai-programmer（AI 程序员）** — 实现 NPC/敌人 AI 行为
- **technical-artist（技术美术）** — 创建 VFX、着色器效果和视觉反馈
- **sound-designer（音效设计师）** — 定义音频事件、打击音效和环境战斗音频
- **qa-tester（QA 测试员）** — 编写测试用例并验证实现

## 角色激活方式

在每个阶段，读取对应的角色文件并扮演该角色：
- `agents/game-designer.md` → 设计机制
- `agents/gameplay-programmer.md` → 实现核心代码
- `agents/ai-programmer.md` → AI 行为
- `agents/technical-artist.md` → 视觉反馈
- `agents/sound-designer.md` → 音频事件
- `agents/qa-tester.md` → 测试验证

始终在每个阶段提供完整上下文（设计文档路径、相关代码文件、约束条件）。使用 `TodoWrite` 跟踪各阶段进度。

## 流水线

### 阶段 1：设计
激活 **game-designer** 角色：
- 在 `design/gdd/` 中创建或更新设计文档，涵盖：机制概述、玩家幻想、详细规则、含变量定义的公式、边缘情况、依赖项、含安全范围的调优旋钮、验收标准
- 使用 AskUserQuestion 呈现设计方案选项
- 输出：完成的设计文档

### 阶段 2：架构设计
激活 **gameplay-programmer** 角色（如涉及 AI，同时参考 **ai-programmer** 角色）：
- 审查设计文档
- 设计代码架构：类结构、接口、数据流
- 识别与现有系统的集成点
- 使用 AskUserQuestion 呈现架构选项
- 输出：架构草图，含文件列表和接口定义

### 阶段 3：实现（顺序执行）
依次激活各角色：
- **gameplay-programmer**：实现核心战斗机制代码
- **ai-programmer**：实现 AI 行为（如果功能涉及 NPC 反应）
- **technical-artist**：创建 VFX 和着色器效果规格
- **sound-designer**：定义音频事件列表和混音说明
- 使用 TodoWrite 跟踪每个子阶段的完成状态

### 阶段 4：集成
- 连接玩法代码、AI、VFX 和音频
- 确保所有调优旋钮已暴露且数据驱动
- 验证功能与现有战斗系统的兼容性

### 阶段 5：验证
激活 **qa-tester** 角色：
- 根据验收标准编写测试用例
- 测试设计文档中记录的所有边缘情况
- 验证性能影响在预算范围内
- 为发现的问题提交缺陷报告

### 阶段 6：签收
- 收集所有阶段的结果
- 报告功能状态：完成 / 需要修改 / 阻塞
- 列出所有未解决问题及其负责人

## 输出
一份总结报告，涵盖：设计完成状态、各阶段的实现状态、测试结果以及任何未解决的问题。

## 协作协议
1. 每阶段开始前读取对应角色文件
2. 使用 AskUserQuestion 呈现关键决策选项
3. 使用 TodoWrite 跟踪进度
4. 写入任何文件前获得用户批准
