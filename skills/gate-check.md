# Gate Check — 阶段关卡验证技能

触发条件:
- 用户说"关卡检查" / "gate check" / "阶段推进" / "就绪检查" / "能否进入下一阶段"
- 用户提供了目标阶段名称

---

此技能验证项目是否已准备好进入下一个开发阶段。它会检查所需工件、质量标准和阻碍项。

**与 project-stage-detect 的区别**：project-stage-detect 是诊断性的（"我们在哪？"）。本技能是规范性的（"我们准备好推进了吗？"并附带正式判定）。

---

## 开发阶段（7 个）

1. **概念** — 头脑风暴、游戏概念文档
2. **系统设计** — 映射系统、编写 GDD
3. **技术搭建** — 引擎配置、架构决策
4. **前期制作** — 原型开发、垂直切片验证
5. **正式制作** — 功能开发
6. **打磨** — 性能优化、试玩测试、缺陷修复
7. **发布** — 上线准备、平台认证

---

## 1. 解析参数

- **带参数**：验证是否准备好进入指定阶段
- **无参数**：使用 project-stage-detect 的启发式方法自动检测当前阶段，然后验证下一个阶段转换

---

## 2. 阶段关卡定义

### 关卡：概念 → 系统设计

**所需工件：**
- [ ] `design/gdd/game-concept.md` 存在且有内容
- [ ] 游戏支柱已定义

**质量检查：**
- [ ] 游戏概念已经过评审
- [ ] 核心循环已描述且已理解
- [ ] 目标受众已确定

### 关卡：系统设计 → 技术搭建

**所需工件：**
- [ ] 系统索引存在于 `design/gdd/systems-index.md`
- [ ] `design/gdd/` 中至少有 1 份完整的 GDD

**质量检查：**
- [ ] GDD 通过设计评审（8 个必需章节齐全）
- [ ] 系统依赖关系已在系统索引中映射
- [ ] MVP 优先级层级已定义

### 关卡：技术搭建 → 前期制作

**所需工件：**
- [ ] 引擎已选定（技术栈不再是占位符）
- [ ] `docs/architecture/` 中至少有 1 份 ADR
- [ ] 引擎参考文档存在于 `docs/engine-reference/`

**质量检查：**
- [ ] 架构决策覆盖核心系统
- [ ] 命名规范和性能预算已设置

### 关卡：前期制作 → 正式制作

**所需工件：**
- [ ] `prototypes/` 中至少有 1 个带 README 的原型
- [ ] `production/sprints/` 中存在首个 Sprint 计划
- [ ] 系统索引中所有 MVP 层级的 GDD 均已完成

**质量检查：**
- [ ] 原型验证了核心循环假设
- [ ] Sprint 计划引用了 GDD 中的真实工作项
- [ ] 垂直切片范围已定义

### 关卡：正式制作 → 打磨

**所需工件：**
- [ ] `src/` 中有按子系统组织的有效代码
- [ ] GDD 中的所有核心机制已实现
- [ ] 主要游玩路径可从头玩到尾
- [ ] 测试文件存在于 `tests/`
- [ ] 至少有 1 份试玩报告

**质量检查：**
- [ ] 测试通过
- [ ] 没有严重/阻断级 Bug
- [ ] 核心循环游玩效果符合设计
- [ ] 性能在预算范围内

### 关卡：打磨 → 发布

**所需工件：**
- [ ] 里程碑计划中的所有功能已实现
- [ ] 内容已完成
- [ ] 本地化字符串已外部化
- [ ] QA 测试计划已存在
- [ ] 平衡数据已经过评审
- [ ] 发布清单已完成
- [ ] 商店元数据已准备（如适用）
- [ ] 更新日志/补丁说明已起草

**质量检查：**
- [ ] 完整的 QA 轮次已签字确认
- [ ] 所有测试通过
- [ ] 性能目标在所有目标平台上均已达成
- [ ] 无已知的严重、高或中等级别 Bug
- [ ] 无障碍基础项已覆盖
- [ ] 所有目标语言的本地化已验证
- [ ] 法律要求已满足
- [ ] 构建编译和打包正常

---

## 3. 执行关卡检查

针对目标关卡中的每一项：

### 工件检查
- 使用 Glob 和 Read 验证文件存在且有实质性内容
- 不要只检查文件是否存在 — 验证文件有真实内容

### 质量检查
- 对于设计评审检查：Read GDD 并检查 8 个必需章节
- 对于性能检查：与性能分析数据或最近的性能目标进行对比
- 对于本地化检查：在 `src/` 中使用 Grep 搜索硬编码字符串

---

## 4. 协作评估

对于无法自动验证的项目，**使用 AskUserQuestion 询问用户**：

- "我无法自动验证核心循环游玩是否良好。是否已进行试玩测试？"
- "未找到试玩报告。是否进行了非正式测试？"
- "性能分析数据不可用。是否要运行性能分析？"

**切勿对无法验证的项目假设为 PASS。** 将其标记为"需要人工检查"。

---

## 5. 输出判定结果

```
## 关卡检查：[当前阶段] → [目标阶段]

**日期**：[日期]
**检查者**：gate-check 技能

### 所需工件：[X/Y 项已就绪]
- [x] design/gdd/game-concept.md — 已存在，2.4KB
- [ ] docs/architecture/ — 缺失（未找到 ADR）
- [x] production/sprints/ — 已存在，1 个 Sprint 计划

### 质量检查：[X/Y 项通过]
- [x] GDD 包含 8/8 个必需章节
- [ ] 测试 — 失败（3 个失败）
- [?] 核心循环试玩 — 需要人工检查

### 阻碍项
1. **无架构决策记录** — 创建一份覆盖核心系统架构的 ADR 后再进入正式制作
2. **3 个测试失败** — 修复失败测试后再推进

### 判定：[通过 / 有顾虑 / 未通过]
- **通过**：所有所需工件齐全，所有质量检查通过
- **有顾虑**：存在少量差距，可在下一阶段中解决
- **未通过**：必须先解决关键阻碍项才能推进
```

---

## 6. 后续行动

根据判定结果，建议具体的下一步操作：

- 没有游戏概念？ → 头脑风暴
- 没有系统索引？ → 系统映射
- 缺少设计文档？ → 逆向文档化
- 缺少 ADR？ → 架构决策
- 测试失败？ → 激活 lead-programmer 或 qa-tester
- 没有试玩数据？ → 试玩报告
- 性能未知？ → 性能分析
- 未本地化？ → 本地化
- 准备发布？ → 上线清单

## 协作协议

1. **先扫描**：检查所有工件和质量关卡
2. **询问未知项**：不要对你无法验证的事项假设为通过
3. **展示发现**：显示完整清单及其状态
4. **用户决定**：判定仅为建议 — 由用户做出最终决定
5. **获取批准**："是否可以将此关卡检查报告写入 `production/gate-checks/`？"

**绝不**阻止用户推进 — 判定仅为建议性意见。记录风险，让用户自行决定。
