# Test Flakiness — 测试不稳定性检测

不稳定测试是那种有时通过有时失败、没有任何代码变更的测试。不稳定测试在某种程度上比没有测试更糟 — 它们训练团队忽略红色CI运行，掩盖真正的失败。

**输出:** 更新 `tests/regression-suite.md` 隔离部分 + 可选报告

---

## 触发条件

- 用户说"测试不稳定" / "test flakiness" / "检测不稳定测试"

---

## 1. 解析参数

**模式:** `[ci-log-path]`(分析特定日志) / `scan`(扫描所有可用CI日志) / `registry`(审查已知隔离测试) / 无参数(自动检测)

---

## 2. 定位 CI 日志数据

GitHub Actions: 检查 `test-results/` 下 `.xml` 文件。本地日志文件: 读取参数指定路径。无日志数据: 提示选项。

---

## 3. 解析测试结果

JUnit XML / 纯文本日志。构建 `test_id → [多次运行结果]` 表。

---

## 4. 识别不稳定测试

不稳定阈值: 高(>25%失败 → 隔离) / 中(5-25% → 调查修复) / 低(1-5% → 监控)。

分类原因: 时序/异步 / 顺序依赖 / 随机种子 / 资源泄漏 / 外部状态 / 浮点数 / 场景加载竞争。

---

## 5. 建议行动

**高不稳定性** → 隔离+修复。**中** → 调查+修复。**低** → 监控。

---

## 6. 生成报告

```
## Flakiness Detection Results
### 不稳定测试表 (测试 | 系统 | 失败率 | 原因 | 建议)
### 清洁测试 (无稳定性问题)
```

---

## 7. 更新回归套件

征求批准后编辑 `tests/regression-suite.md` 隔离部分。可选写入完整报告 `production/qa/flakiness-report-[date].md`。

Verdict: **COMPLETE** — 不稳定报告已写入。

---

## 协作协议

- **绝不删除测试文件** — 隔离意味着标注+列出，不是删除
- **统计置信度重要** — 少于3次运行的发现标记为"suspected"
- **修复始终是目标** — 隔离是临时的
- **写入前询问** — 两次写入都需明确批准
