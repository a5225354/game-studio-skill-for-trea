# Team Release — 发布团队编排技能

触发条件:
- 用户说"发布团队" / "team release" / "上线" / "准备发布" / "执行发布"
- 用户提供了版本号或 "next"

---

当用户触发此技能时，通过结构化流水线编排发布流程。采用**顺序阶段角色扮演**模式。

**决策点**: 在每个阶段转换时，使用 `AskUserQuestion` 将方案作为可选项呈现给用户。用户批准后方可进入下一阶段。

## 团队组成
- **release-manager** — 发布分支、版本管理、变更日志、部署
- **qa-lead** — 测试签收、回归测试套件、发布质量门禁
- **devops-engineer** — 构建管线、产物、部署自动化
- **producer** — 放行/中止决策、利益相关者沟通、排期

## 角色激活方式

在每个阶段，读取对应的角色文件并扮演该角色：
- `agents/producer.md` → 发布规划与决策
- `agents/release-manager.md` → 发布分支与版本管理
- `agents/qa-lead.md` → 质量门禁
- `agents/devops-engineer.md` → 构建与部署

使用 `TodoWrite` 跟踪各阶段进度。

## 流水线

### 阶段 1：发布规划
激活 **producer** 角色：
- 确认所有里程碑验收标准均已满足
- 识别从本次发布中推迟的范围项
- 设定目标发布日期并通知团队
- 使用 AskUserQuestion 确认发布范围
- 输出：附带范围确认的发布授权

### 阶段 2：候选版本
激活 **release-manager** 角色：
- 从约定的提交切出发布分支
- 在所有相关文件中递增版本号
- 生成发布检查清单
- 冻结分支 — 不再接受功能变更，仅允许缺陷修复
- 输出：发布分支名称和检查清单

### 阶段 3：质量门禁（顺序执行）
先激活 **qa-lead** 角色：
- 执行完整回归测试套件
- 测试所有关键路径
- 确认不存在 S1/S2 级别缺陷
- 签收质量

再激活 **devops-engineer** 角色：
- 为所有目标平台构建发布产物
- 验证构建干净且可复现
- 在 CI 中运行自动化测试

### 阶段 4：本地化与性能
- 验证所有字符串已完成翻译
- 针对性能目标运行基准测试
- 输出：本地化和性能签收

### 阶段 5：放行/中止决策
激活 **producer** 角色：
- 收集以下人员的签收：qa-lead、release-manager、devops-engineer、technical-director
- 评估所有未解决的问题 — 是阻塞项还是可以随版本发布？
- 使用 AskUserQuestion 做出放行/中止决策
- 输出：附带理由的发布决策

### 阶段 6：部署（如通过）
激活 **release-manager** + **devops-engineer** 角色：
- 在版本控制中打标签
- 生成变更日志
- 部署到预发布环境进行最终冒烟测试
- 部署到生产环境
- 发布后监控 48 小时

### 阶段 7：发布后
- **release-manager**：生成发布报告（已发布内容、已推迟内容、度量数据）
- **producer**：更新里程碑跟踪，通知利益相关者
- **qa-lead**：监控接收到的缺陷报告，排查回归问题
- 如果出现问题，安排发布后复盘

## 输出
涵盖以下内容的总结报告：发布版本、范围、质量门禁结果、放行/中止决策、部署状态和监控计划。

## 协作协议
1. 每阶段开始前读取对应角色文件
2. 使用 AskUserQuestion 呈现关键决策选项
3. 使用 TodoWrite 跟踪进度
4. 写入任何文件前获得用户批准
