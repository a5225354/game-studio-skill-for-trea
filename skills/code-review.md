# Code Review — 架构级代码审查技能

触发条件:
- 用户说"代码审查" / "review code" / "code review" / "审查代码"
- 用户指定了文件或目录路径

---

## 协作协议

**代码审查是协作式的，不是审计式的。** 你的目标是提升代码质量，同时尊重用户的设计选择。

---

## 审查流程

### 1. 确定审查范围

如果用户提供了具体文件路径，审查该文件。如果用户提供了目录，使用 Glob 搜索该目录下的源文件并列出供用户选择。

如果用户未提供范围，使用 AskUserQuestion 询问:
> "你想审查哪个文件或目录的代码？"

### 2. 读取审查目标

读取目标文件。同时读取相关的:
- 对应的设计文档（在 `design/gdd/` 中）
- 对应的架构决策记录（在 `docs/architecture/` 中）
- 该路径适用的编码规范（在 `rules/` 中）

### 3. 执行审查

按以下五个维度进行审查：

#### A. 架构合规性
检查代码是否遵循项目架构约定:
- ✓ 是否符合对应的 ADR（架构决策记录）？
- ✓ 是否遵循该路径的编码规范？
- ✓ 模块边界是否正确？
- ✗ 是否有跨领域的耦合？

#### B. SOLID 原则
- **单一职责**: 类/函数是否只做一件事？
- **开闭原则**: 扩展是否容易而不修改现有代码？
- **里氏替换**: 子类是否可替换父类？
- **接口隔离**: 接口是否精简且专注？
- **依赖反转**: 是否依赖抽象而非具体实现？

#### C. 数据驱动与可配置性
- ✓ 游戏数值是否来自配置/数据文件（非硬编码）？
- ✓ 调优旋钮是否已暴露？
- ✓ 时间计算是否使用 delta time？

#### D. 可测试性
- 公共方法是否可单元测试？
- 依赖项是否可模拟/注入？
- 逻辑是否与表现层分离？

#### E. 性能
- 热路径中是否有不必要的分配？
- 是否有明显的性能瓶颈？
- 是否符合性能预算？

### 4. 呈现审查结果

以结构化的格式呈现：

```
正在审查 [文件路径]...

**架构合规性**：
✓ 符合 ADR-003（无状态工具模式）
✓ 数据驱动值（gameplay-code 规则）
✗ 问题：直接引用 [类型] — 破坏了依赖注入

**SOLID 原则**：
✓ 单一职责 — 只负责 [职责]
✗ 依赖反转 — 与 [类型] 紧密耦合

**可测试性**：
✗ 难以模拟 [依赖] 

**性能**：
✓ 热路径中无分配
⚠ 警告：每次调用进行字典查找 — 考虑缓存

**建议**（按优先级排列）：
1. 高优先级：[问题描述] + [建议修复方案]
2. 中优先级：[问题描述] + [建议修复方案]
3. 低优先级：[优化建议]
```

### 5. 协作修复

对高优先级问题，展示修复方案：

> "问题 #1 可以通过 [方案] 解决。这个改动让你可以 [收益]。我可以展示重构后的代码吗？"

用户确认后展示重构代码，获得批准后才能编辑文件。

### 6. 编码规范检查

根据被审查文件所在的路径，对照对应的编码规范进行检查：

| 文件路径 | 检查的规范 |
|---------|-----------|
| `src/gameplay/**` | 数据驱动值、delta time、禁止直引 UI、状态机转换表 |
| `src/core/**` | 零分配热路径、线程安全、API 稳定性 |
| `src/ai/**` | 性能预算、可调试性、数据驱动参数、预警机制 |
| `src/networking/**` | 服务器权威、版本化消息、安全性 |
| `src/ui/**` | 不持有游戏状态、本地化就绪、无障碍 |
| `src/tools/**` | 不依赖游戏代码、类型安全 |
| `tests/**` | 测试命名、Arrange/Act/Assert、无外部依赖 |

### 7. 与设计文档的一致性

如果存在对应的 GDD，检查代码实现是否与设计文档一致:
- 公式实现是否匹配设计文档中的公式？
- 边界情况处理是否按文档要求？
- 调优旋钮是否正确暴露？

## 护栏

- 审查是建议性的，不是命令式的
- 如果代码在原型目录下（`prototypes/`），应用宽松标准
- 提供具体修复方案而不只是指出问题
- 在编辑任何文件前获得用户批准
- 如果用户不同意某个建议，尊重用户的决策并继续
